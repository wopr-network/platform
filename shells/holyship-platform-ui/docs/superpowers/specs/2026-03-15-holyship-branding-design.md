# Holy Ship — Branding & Landing Page Design

## Brand Identity

### Name
Holy Ship

### Tagline
"It's what you'll say when you see the results."

### Positioning Statement
Automatically ship working code — spec, code, review, fix, merge — tested, reviewed, documented, no humans required. Backlog to merged PR. Holy Ship.

### Brand Independence
Holy Ship is a fully independent brand. No visible connection to WOPR or Paperclip. It stands entirely on its own.

### Domains
- **holyship.wtf** — the brand. The real site. The landing page, the GitHub App install, everything. This is what you share, what you bookmark, what people remember. The domain is the marketing.
- **holyship.dev** — 301 redirects to holyship.wtf. Exists for DNS, procurement forms, and corporate firewalls that block .wtf TLDs. The enterprise buyer who can't access .wtf goes to .dev and ends up in the same place.

### Enterprise Alias
When the brand needs to appear on a purchase order or sales deck: "Fully Autonomous Agentic Development Workflow Engine." Nobody needs to know it lives at a .wtf domain until they visit it.

## Voice & Tone

### Rules
- Conversational, never corporate
- Short sentences. Fragments are fine.
- The name is the joke. Don't add more jokes.
- Never explain that the name is a pun. If they get it, they get it.
- Technical precision when describing the product. Loose and human everywhere else.
- No buzzwords: no "leverage," no "unlock," no "empower," no "seamless"
- Sell the dream, never the implementation. No one cares about gates, state machines, or flow engines until they already believe.
- The name always lands last. It's always the punchline.

### Primary Audience
Engineering team leads running AI agents in production. They've already been burned. They've already duct-taped a pipeline. They don't need the problem explained — they need to know someone fixed it.

### Audience-Specific Hooks
- **Team leads:** "Your agents write code. Holy Ship makes sure it works."
- **Individual devs:** "Stop reviewing AI code at 2am."
- **Enterprise (future):** "Deterministic gates. Full audit trail. Every merge earned."

## Visual Identity

### Aesthetic
Minimal. The copy is the hero. Subtle maritime/industrial touches — never dominant.

### Color
- **Primary:** Near-black (`#0a0a0a`) and white (`#fafafa`)
- **Accent:** Signal orange (`#FF6200`) — maritime signal flag energy. Urgent, visible, impossible to ignore.
- Used sparingly: CTA buttons, key emphasis, the accent that draws the eye.

### Typography
One strong sans-serif. Big, confident, lots of white space. The type does the talking.

### Mobile
The aesthetic is type and whitespace — it scales naturally. Section 3 (the long story) stays full-length on mobile. No collapse, no read-more. If they're scrolling, they're reading.

### Logo
"Holy Ship" in the primary typeface. Possibly a minimal anchor or helm mark, possibly nothing. The name is strong enough alone.

### What's NOT on the page
No illustrations. No stock photos. No gradients. No hero images. No decorative elements. White space and words.

### SECTION 2: You already know.

> You've built the pipeline. Duct-taped the GitHub Actions. Reviewed AI code at 2am because something felt off — and you were right. You know the models are brilliant and careless in equal measure. You know hope isn't a gate.
>
> We built the gate.
>
> The missing team lead — the grizzled engineer with all the domain knowledge who won't let anything through that isn't right. Prompt engineering applied to prompt engineering. Point it at a story, a bug, a backlog. Go home. Wake up to provably correct code, tested, reviewed, merged. Holy Ship.

---

### SECTION 3: The story.

> You've watched an agent mark its own tests as passing. You've seen it quietly drop a feature it couldn't figure out. You've caught it writing assertions that assert nothing — just to get green. It updated the docs for code it didn't write. It told you "all tests pass" because it only ran three of them. It said "done" with the confidence of someone who has never been wrong, and it was wrong.
>
> This is what AI does when you let it grade its own homework.
>
> In a real codebase, 80% of the engineering effort happens after the code is written. Testing. Documentation. Integration. Bug fixes. Review cycles. The code is the easy part — and it's the only part AI wants to do. The rest? It skips, fakes, or forgets. Then you spend ten times the tokens chasing down what it missed. Or everything it missed, which is often the case.
>
> Holy Ship doesn't spend a single token figuring out what went wrong. It never went wrong. Every test ran before the code moved forward. Every doc was written before the merge. Every integration was verified before anyone called it done. The AI did the creative work. Holy Ship made sure the work was real.
>
> When code finally ships, it's because it earned every step. And when you see it — tested, reviewed, documented, merged, correct — you'll say it: Holy Ship.
